## How to Become a Maintainer

Contributors who have demonstrated:

1. **Code Quality**: Consistent high-quality contributions
2. **Community**: Helping others in issues and discussions
3. **Documentation**: Improving documentation
4. **Testing**: Adding comprehensive tests
5. **Review**: Reviewing others' PRs

Can be nominated by existing maintainers.

## Maintainer Responsibilities

### Code Review

- Review PRs in a timely manner
- Ensure code quality and test coverage
- Enforce project standards

### Releases

- Follow the release process in RELEASING.md
- Ensure all checks pass before release
- Document changes properly

### Community

- Respond to issues and discussions
- Help users with questions
- Moderate community interactions

### Security

- Review security reports promptly
- Coordinate security fixes
- Follow responsible disclosure

## Decision Making

- **Day-to-day**: Individual maintainers can merge reviewed PRs
- **Breaking Changes**: Require discussion and consensus
- **Architecture**: Major changes require RFC process

## RFC Process

For significant architectural changes:

1. Open an RFC issue with `RFC:` prefix
2. Allow at least 1 week for community feedback
3. Address concerns and iterate
4. Final approval from 2+ maintainers
